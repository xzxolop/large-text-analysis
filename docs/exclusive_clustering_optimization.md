# Оптимизация exclusive clustering (TF‑IDF × log(freq))

Документ фиксирует изменения, сделанные для ускорения **exclusive clustering** без ухудшения качества (метрика и логика выбора «лучшего слова» сохранены).

## Контекст

В проекте есть два подхода:

- **PMI‑кластеризация**: считается по запросу пользователя (обычно быстро).
- **Exclusive clustering**: каждому предложению назначается одно «наиболее релевантное» слово по метрике \(TF\text{-}IDF \times \log(freq)\).  
  Проблема: при «раскрытии»/итеративных переходах пользователю приходилось ждать до ~10 секунд.

## Цель

Сделать exclusive clustering сопоставимой по отзывчивости с PMI:

- убрать пересчёты TF‑IDF при каждом действии;
- убрать лишние тяжёлые операции внутри кластеризации;
- добавить кэширование там, где результаты переиспользуются.

## Что сделали (хронология)

### 1) Ускорили базовую exclusive‑кластеризацию в `ExclusiveClusterer`

**Было**: в реализации на каждое предложение делались лишние операции (в том числе сортировка кандидатов), что на больших корпусах давало значимую задержку.

**Стало**:

- кластеризация переписана на быстрый проход по CSR‑матрице TF‑IDF;
- вместо полной сортировки кандидатов в строке — поиск максимума (argmax) по скору \(TF\text{-}IDF \times \log(freq)\).

Файл: `src/analysis/exclusive_clusterer.py`

### 2) Убрали пересборку `SearchEngine` и TF‑IDF в `iterative_exclusive_clustering`

**Было**: `SearchEngine.iterative_exclusive_clustering()` создавал временные `SearchEngine` для каждого шага, что приводило к повторному `TF-IDF fit_transform` на подмножествах предложений — самое дорогое место.

**Стало**:

- итеративное сужение переписано так, чтобы работать **на исходной TF‑IDF матрице**, не создавая временных движков;
- добавлена кластеризация по подмножеству индексов (`cluster_on_indices`), чтобы эффективно обрабатывать «текущие» предложения.

Файл: `src/search/search_engine.py`

### 3) Добавили кэш «лучшее слово для каждого предложения»

Это самый большой выигрыш для UX.

**Идея**: для каждого предложения заранее (лениво) сохраняем «победителя» — индекс слова, которое максимизирует \(TF\text{-}IDF \times \log(freq)\).

**Стало**:

- добавлен массив `best_feature_idx[doc]` (или `-1`, если предложение пустое);
- при вызовах exclusive clustering **без исключений** (`excluded_words=None`) результаты собираются из кэша, без обхода всех ненулевых значений sparse‑строк.

Файл: `src/analysis/exclusive_clusterer.py`

## Почему качество не пострадало

Во всех пунктах выше:

- метрика осталась той же: \(score = TF\text{-}IDF(word, doc) \times \log(freq(word)+1)\);
- правило выбора слова осталось тем же: берём слово с **максимальным score** в каждом предложении;
- изменения касаются только производительности (как быстро вычислить тот же результат).

## Текущее поведение и заметки

- **Обычная exclusive‑кластеризация** (без `excluded_words`) теперь максимально быстрая за счёт кэша.
- **С `excluded_words`** по‑прежнему выполняется выбор лучшего не‑исключённого слова (это важно для корректности итеративного финального шага). Этот режим можно ускорять отдельно (например, через top‑K кандидатов + fallback), но это не входило в текущий этап.
